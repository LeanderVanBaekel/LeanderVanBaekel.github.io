Workflow

In dit artikel schrijf ik over de dingen die ik geleerd heb in het afgelopen half jaar tijdens mijn minor Everything Web.

Het begin

Het is zo ver. Je krijgt een opdracht, een project die je van het begin tot het eind moet ontwikkelen. Maar waar begin je nou eigenlijk?

Dit is iets waar ik erg veel moeite mee had in het begin. Begin je meteen met het schrijven van code? of ga je toch eerst een ontwerp maken? in de afgelopen projecten heb ik het allebei geprobeerd en mijn voorkeur gaat uit naar een ontwerp maken. Dit hoeft niet een ontwerp in de traditionele zin te zijn zoals schetsen en in illustrator werken. Maar het is belangrijk dat je voor jezelf opschrijft wat je precies wil gaan doen voor het project. Een van de eerste vragen die je jezelf kan stellen is: Wat zijn de belangrijkste onderdelen/features voor een minimaal werkende site/app? Wat zijn features die ik er nog graag bij zou willen hebben?

Als je hier een lijst van maakt kun je verder met: hoeveel tijd gaat het mij ongeveer per onderdeel kosten? Op deze manier heb je een ruwe schatting hoeveel tijd je aan het project besteed en welke features je dus waarschijnlijk wel en niet kunt uitwerken.

Ik vind het zelf fijn om vanaf hier wat schetsen op papier te maken om alvast wat layouts uit je hoofd te krijgen. Als ik dit niet doe ga ik meteen beginnen met het eerst wat in me op komt, maar dit is niet altijd perse het beste. Als ik ga schetsen voel ik snel of een layout gaat werken of niet en kun je makkelijk schuifen met onderdelen.

Een belangrijke fout die ik gemaakt heb is het niet vast stellen van een database structuur. Als je aan een nieuw project begint waar nog geen database van is, is het goed om deze uit te tekenen zodat je voor je ziet hoe de verbindingen precies lopen. Als je al in de code bezig bent kun je namelijk snel het overzicht verliezen.

Kunnen we nu echt beginnen?

Na het bedenken van het datamodel begin ik te werken in HTML. De (door Krijn) opgelegde PE (progressive enhancement) regels schrijven voor dat je eerst alle HTML structuren vast legt. Echter gaat het vaak al samen een begin van je CSS layout. Hiermee kun je het best altijd mobile-first beginnen. Op deze mannier krijg je later geen last van ruimte te kort omdat je scherm kleiner wordt maar kun je juist dingen ruimer opzetten voor grotere schermen. In dit proces ga ik ook opzoek naar een passend font, kleuren en vormen als die niet door de opdracht gever gesteld zijn.

Op dit moment vraag je je misschien af of ik niet ben vergeten om een framework/preprocessor/git-brench/server op te zetten. Nee, dat ben ik niet. De belangrijkste structuren heb je nu in de HTML en CSS gedefinieerd. Nu kun je aan de hand van de features die je nodig hebt gaan kijken welk circus je uit de kast moet gaan trekken.

In mijn geval zou ik voor Node.js gaan om dynamische data serverside te kunnen renderen zodat de website/app niet afhankelijk is van javascript.

Agile

Vanaf hier gaat het proces meer richting iteraties. Het lijstje afwerken met features die ik wil implementeren om de MVP (minimum viable product) te creëren.

Ik vind het fijn om met Github te werken. Hierdoor ontstaan er vanzelf backups van alles wat je schrijft. mocht je spijt krijgen en iets terug willen wat je veranderd hebt kan dat. Ook is het handig om met feature-branches te werken om zo in te kaderen voor jezelf waar je mee aan het werk bent.

Deployment

Voor dat het project af is ga ik vast nadenken over hoe ik de site/app wil hosten. Dit moet je niet tot het laatste moment uit stellen want het is altijd meer werk dan je denkt, en er gaat altijd wel iets fout. Zorg dus dat je vast een test draait met een nog niet versie die nog niet helemaal af is. Zo weet je tegen welke problemen je aan kan gaan lopen tijdens de echte deployment.

Conclusie

Een project loopt nooit helemaal volgens plan, maar als ik mijzelf aan deze richtlijnen hou heb ik een rode draad voor het project. Ik hoop dat jij er ook wat aan hebt. Al is het maar dat je nu denkt: zo zou ik dus nooit te werk gaan! ;)
